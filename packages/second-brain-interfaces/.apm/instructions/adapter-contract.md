# Adapter Contract

This document defines how an adapter translates second-brain capability
calls into concrete transport operations. The interface layer is
transport-neutral -- it owns envelopes and capability IDs, not HTTP methods,
Copilot APIs, or GitHub Issue payloads.

## Capability-to-transport mapping

An adapter MUST:

1. Accept a typed request envelope (see the capability's `schema/request.schema.json`).
2. Validate the envelope against the schema before dispatching. Return
   `VALIDATION_ERROR` immediately if validation fails; do not forward an
   invalid request to the provider.
3. Map the `capability` field to the provider's registered transport entry
   (see `schemas/provider-declaration.schema.json`, `capabilities[].id`).
4. Attach `correlation_id` to the outbound transport message exactly as
   received; do not mutate or regenerate it.
5. Return a typed response envelope (the capability's `schema/response.schema.json`)
   or the standard error envelope (`schema/error.schema.json`) -- never
   raw transport payloads.

## Correlation ID flow

```
client                adapter               provider
  |                      |                      |
  |-- request (cid) ---> |                      |
  |                      |-- transport(cid) --> |
  |                      |                      | (synchronous or async)
  |                      | <-- response(cid) --|
  | <-- response(cid) --|                      |
```

- `correlation_id` is a UUID v4 generated by the client per request.
- Adapters MUST pass `correlation_id` through unchanged in both directions.
- For asynchronous providers (`interaction_mode: "asynchronous"`) the
  adapter issues the transport call, then polls or awaits a callback
  keyed on `correlation_id` before returning the response envelope to
  the client. The polling strategy is adapter-specific.

## Error code conventions

| Code               | When to use                                                          |
|--------------------|----------------------------------------------------------------------|
| `VALIDATION_ERROR` | Request envelope fails schema validation.                            |
| `NOT_FOUND`        | Provider cannot locate the requested target (e.g. forget target).   |
| `UNAUTHORIZED`     | Adapter lacks permission to reach the provider.                      |
| `PROVIDER_ERROR`   | Provider returned an unrecoverable error.                            |
| `TRANSIENT`        | Timeout, rate limit, or temporary unavailability -- safe to retry.  |
| `UNKNOWN`          | Catch-all for unmapped errors.                                       |

Adapters MUST NOT surface transport-layer error codes (e.g. HTTP 4xx/5xx
status integers) directly to the client. Map them to the codes above.

## Transport neutrality

This interface layer defines:
- Capability identifiers (`second-brain.think.v1`, etc.)
- Request and response envelope schemas
- Error code vocabulary
- Correlation ID semantics

This interface layer does NOT define:
- How the transport encodes the envelope (JSON body, GitHub Issue
  template, Copilot cross-session message, function argument, etc.)
- Authentication mechanisms
- Rate-limiting policies
- Retry intervals
- Provider-specific configuration

All of the above are the responsibility of the concrete adapter
implementation and the provider declaration's `transport.config` field.

## Provider declaration lookup

Before dispatching, an adapter resolves the provider via a provider
declaration (see `schemas/provider-declaration.schema.json`). The
declaration supplies:

- Which capabilities the provider supports
- The `interaction_mode` (synchronous vs asynchronous)
- The `transport.type` to use
- The `knowledge_roots` available for scoped retrieval

Adapters MUST reject requests for capabilities not listed in the
provider's declaration rather than silently forwarding them.
